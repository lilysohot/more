# 三 Agent 深度辩论 + 汇总 Agent 方案报告

**文档版本**：v1.1  
**编写日期**：2026-04-23  
**最近审定日期**：2026-04-23  
**适用项目**：公司分析研报助手 / 小财大用（MoreMoney）

---

## 一、方案目标

当前需求文档中的“三维合一投资决策委员会框架”已经明确了三个核心分析视角：

1. 芒格视角
2. 产业专家视角
3. 审计专家视角

现计划将原本一次性生成的“三维分析”升级为多 Agent 协作架构：

1. 由 3 个独立 Agent 分别产出各自立场报告
2. 增加 1 个汇总 Agent，对三份报告进行交叉整合
3. 由汇总 Agent 输出统一结论、评分、风险提示和最终可编排报告
4. 后续再由报告生成层输出 Markdown / HTML

该方案的核心价值是：

1. 提高分析结构清晰度
2. 增强不同视角之间的“辩论感”和“证据链”
3. 降低单次 Prompt 混合多角色时的角色串扰
4. 为后续引入权重、冲突分析、可解释性评分打下基础

---

## 二、结论摘要

### 2.1 总体结论

该方案方向正确，技术上可行，适合作为现有系统的下一阶段演进方向。

但本次架构复核后的判断不是“可直接按文档进入完整开发”，而是：

1. 作为多 Agent 工作流 MVP，可行
2. 作为正式上线版本，仍需补齐若干前置条件
3. 最大风险不在 Agent 数量本身，而在任务可靠性、结构化解析稳定性和基础数据质量

### 2.2 可行性判断

1. 从产品逻辑看：高度契合需求文档中“三维深度辩论”的原始设计
2. 从当前代码看：后端已有分析任务、报告生成、LLM 调用、活跃任务追踪等基础设施，可承接升级
3. 从实现成本看：中等，不需要推翻现有系统，可基于现有 `AnalysisService` 渐进式改造
4. 从风险看：主要风险不在架构本身，而在数据质量、Token 成本、响应时长、任务中断恢复和汇总一致性

### 2.3 架构审定后的关键结论

本次复核新增以下判断：

1. 当前系统更适合先完成“多 Agent 工作流骨架验证”，不适合直接承诺“高准确率投研结论”
2. 任务执行可靠性必须前置考虑，不能等多 Agent 做完后再补
3. 结构化输出不应停留在“定义 Schema”，必须同步定义严格 JSON 输出、服务端校验、失败重试和版本记录机制
4. 状态设计应区分“业务状态”和“细粒度阶段”，避免前后端长期耦合
5. 数据质量应被视为上线门槛，而不是普通风险项

### 2.4 推荐落地策略

建议采用：

1. 第一阶段：可观测的串行多 Agent MVP
   - 顺序调用 3 个角色 Agent
   - 最后调用 1 个汇总 Agent
   - 先打通完整链路
   - 同时补齐结构化解析和审计日志
2. 第二阶段：结构化输出和前端产品化
   - 三个 Agent 输出统一 JSON Schema
   - 汇总 Agent 基于结构化数据生成最终结论
   - 增加角色结果查询、分歧展示和失败重试
3. 第三阶段：可靠性与能力增强
   - 将长链路任务迁移到更稳的任务执行机制
   - 增加证据强度、数据来源评分、分歧标注
   - 增加互相质询、冲突点复核或更多专业 Agent

## 四、目标架构设计

### 4.1 目标角色

建议定义以下 4 个 Agent：

1. `munger_agent`
   - 负责护城河、资本配置、管理层诚信、估值安全边际
   - 偏长期主义、企业本质、竞争优势
2. `industry_agent`
   - 负责行业格局、供需变化、成本曲线、周期位置、竞争结构
   - 偏产业链与行业逻辑
3. `audit_agent`
   - 负责财务质量、现金流、关联交易、资产质量、利润真实性
   - 偏风控、财务真实性、报表可信度
4. `synthesis_agent`
   - 汇总前三者输出
   - 负责冲突归纳、共识抽取、最终评分、投资建议、报告编排

### 4.2 建议执行流程

```text
用户输入公司名称/股票代码
    ↓
数据采集与标准化
    ↓
财务指标计算与数据清洗
    ↓
构建统一分析上下文 Context
    ↓
┌──────────────────────────────────────┐
│ 1. munger_agent                      │
│ 2. industry_agent                    │
│ 3. audit_agent                       │
└──────────────────────────────────────┘
    ↓
聚合三份角色报告
    ↓
synthesis_agent 汇总与裁决
    ↓
生成标准 Markdown 报告
    ↓
生成 HTML 报告与图表编排
    ↓
返回用户并保存历史记录
```

### 4.3 新增架构原则

为避免方案落地时偏离工程现实，建议增加以下原则：

1. 先把多 Agent 当作“结构化工作流”，而不是“自由对话系统”
2. 先保证每个阶段可观测、可追踪、可降级，再追求更多 Agent 和更复杂辩论
3. 汇总 Agent 必须具备明确裁决边界，不能只是机械拼接
4. 若基础数据缺失或失真，应优先暴露“不足以判断”，而不是继续生成看似专业的结论

## 五、技术文档摘要

### 5.1 后端模块改造建议

当前仓库已有：

1. `backend/app/services/analysis.py`
2. `backend/app/services/llm_service.py`
3. `backend/app/services/report_generator.py`
4. `backend/app/api/analyses.py`

建议新增或重构如下模块：

```text
backend/app/services/agents/
├── base.py
├── munger_agent.py
├── industry_agent.py
├── audit_agent.py
├── synthesis_agent.py
├── orchestrator.py
└── schemas.py
```

职责划分：

1. `base.py`
   - Agent 基类
   - 统一 prompt 调用、输出解析、重试和日志逻辑
2. `munger_agent.py`
   - 芒格视角分析逻辑
3. `industry_agent.py`
   - 产业专家分析逻辑
4. `audit_agent.py`
   - 审计专家分析逻辑
5. `synthesis_agent.py`
   - 汇总与裁决逻辑
6. `orchestrator.py`
   - 多 Agent 编排和阶段调度
7. `schemas.py`
   - 统一定义 Agent 输入输出结构

### 5.2 建议的数据结构

建议三位角色 Agent 统一输出结构化结果，而不是只返回长文本。

角色 Agent 输出建议：

```json
{
  "role": "munger",
  "summary": "一句话结论",
  "score": 7.5,
  "thesis": ["核心观点1", "核心观点2"],
  "positives": ["优势1", "优势2"],
  "risks": ["风险1", "风险2"],
  "evidence": [
    {
      "item": "ROE 连续三年高于行业均值",
      "source": "年报 2025",
      "source_type": "official",
      "source_date": "2025-12-31",
      "confidence": 0.8
    }
  ],
  "red_flags": ["红旗信号1"],
  "questions": ["仍需验证的问题1"]
}
```

汇总 Agent 输出建议：

```json
{
  "company_profile": "一句话定义公司本质",
  "consensus": ["三方共识1", "三方共识2"],
  "disagreements": [
    {
      "topic": "行业景气持续性",
      "munger": "谨慎乐观",
      "industry": "强烈看多",
      "audit": "中性"
    }
  ],
  "final_score": 7.2,
  "investment_decision": "持有",
  "insufficient_data": false,
  "core_reasons": ["理由1", "理由2"],
  "major_risks": ["风险1", "风险2"],
  "report_sections": {
    "intro": "...",
    "munger_view": "...",
    "industry_view": "...",
    "audit_view": "...",
    "synthesis": "..."
  }
}
```

### 5.3 Prompt 设计原则

1. 三个角色必须使用同一份基础数据上下文
2. 角色职责边界必须明确
3. 输出结构必须固定
4. 必须要求引用“证据”和“未验证点”
5. 汇总 Agent 不应简单拼接，而应执行“归纳 + 冲突协调 + 裁决”
6. 必须要求严格输出 JSON，不允许额外说明文字混入结构化结果
7. 每个 Prompt 需要记录 `prompt_version`，便于回溯和比较

### 5.4 与现有后端流程的衔接方式

建议将现有 `run_analysis` 流程扩展为：

```text
Step 1: 数据采集
Step 2: 财务指标计算
Step 3: 构建统一上下文
Step 4: 执行 munger_agent
Step 5: 执行 industry_agent
Step 6: 执行 audit_agent
Step 7: 执行 synthesis_agent
Step 8: 生成 Markdown/HTML 报告
Step 9: 入库
```

### 5.5 任务执行可靠性要求

当前仓库已有后台异步执行能力，但多 Agent 长链路对可靠性要求更高。

因此建议：

1. MVP 阶段可继续使用现有后台任务机制验证链路
2. 进入产品化阶段前，应补齐更稳的任务执行机制或恢复机制
3. 至少支持以下能力：
   - 单个 Agent 独立失败和记录
   - 汇总阶段独立重试
   - 服务中断后的状态恢复或人工重驱动
   - 任务超时、阶段耗时和错误归因可观测

## 六、可行性分析

### 6.1 技术可行性

结论：高

原因：

1. 当前系统已具备 LLM 调用能力
2. 已具备分析任务和报告生成基础链路
3. 多 Agent 本质上是“多次角色化 LLM 调用 + 汇总调用”
4. 不依赖特殊中间件，完全可以先在现有 FastAPI 服务内实现 MVP

### 6.2 产品可行性

结论：高

原因：

1. 完全符合原始需求文档中的“三维深度辩论”
2. 输出会比单次混合 Prompt 更清楚
3. 用户更容易理解“为什么得出这个结论”
4. 有利于形成差异化产品能力

### 6.3 工程可行性

结论：中高

原因：

1. 现有项目结构支持渐进式改造
2. 后端服务层可扩展
3. 前端报告页可扩展
4. 主要工作量集中在：
   - Prompt 设计
   - 输出结构化
   - 任务状态拆分
   - 报告模板升级
   - 运行记录与恢复机制

### 6.4 性能可行性

结论：中等，可优化

多 Agent 会带来以下成本：

1. Token 消耗增加
2. 总耗时增加
3. API 成本上升
4. 超时和限流概率上升

应对策略：

1. 第一阶段先串行执行，便于调试
2. 后续对三个角色 Agent 做异步并发
3. 控制每个 Agent 输出长度
4. 使用结构化 JSON，减少冗余长文本
5. 对公共上下文进行裁剪
6. 为汇总 Agent 提供“摘要版角色输出”，不要重复灌入全部长文

### 6.5 数据可行性

结论：当前中等偏低，必须设门槛

原因：

1. 当前主流程仍存在 mock 数据和 mock ratios 逻辑
2. 这意味着现阶段更适合验证工作流，不适合验证分析结论准确率
3. 多 Agent 不会自动修复基础数据质量问题，反而可能放大错误表达

因此建议将数据质量提升为上线门槛：

1. 核心财务字段必须来自真实数据源
2. 关键结论必须可追溯到来源和日期
3. 数据不足时必须显式降级为“不足以判断”

## 七、主要风险与应对

### 7.1 风险一：三 Agent 输出风格不一致

问题：不同角色可能篇幅差异很大，导致汇总困难。

应对：

1. 强制统一输出 Schema
2. 控制字段长度
3. 使用明确的 role-specific prompt 模板
4. 服务端做 schema 校验，不通过则进入修复重试

### 7.2 风险二：汇总 Agent 变成机械拼接

问题：如果汇总 Prompt 不好，最终只是在拼接三份报告，价值不高。

应对：

1. 要求汇总 Agent 输出“共识、分歧、裁决”
2. 增加“冲突点分析”字段
3. 让汇总 Agent 必须给出最终投资决策及理由
4. 明确裁决规则，不把最终分数完全交给自由生成

### 7.3 风险三：总耗时过长

问题：3 个角色 + 1 个汇总，可能明显变慢。

应对：

1. 先做 MVP 串行版本
2. 第二阶段改为并发执行三个角色 Agent
3. 为每个 Agent 设置超时与失败降级机制
4. 允许“部分结果成功”时继续生成报告

### 7.4 风险四：垃圾进垃圾出

问题：当前仓库中分析服务仍有 mock 数据逻辑，如果基础数据不真实，多 Agent 只会放大错误。

应对：

1. 先优先完成真实数据采集链路
2. 对每条关键结论标注数据来源和日期
3. 引入置信度字段
4. 对缺失数据明确提示“基于不完整数据推断”
5. 将数据真实性作为上线门槛之一

### 7.5 风险五：评分标准不统一

问题：三个 Agent 打分口径不同，汇总会失真。

应对：

1. 统一 0-10 分量表
2. 明确定义每个分数区间含义
3. 汇总 Agent 使用统一权重规则
4. 后续可配置权重，例如：
   - 芒格 35%
   - 产业 35%
   - 审计 30%

### 7.6 风险六：任务链路不可靠

问题：多 Agent 长链路对中断恢复、超时、重试和多阶段观测提出更高要求。

应对：

1. MVP 阶段补齐阶段级日志和运行记录
2. 产品化阶段引入更稳的任务执行机制或恢复方案
3. 对每个 Agent 执行结果单独持久化
4. 支持单阶段重试，而不是整单重跑

## 八、推荐实施方案

### 8.1 MVP 版本

目标：先跑通 4 Agent 链路，验证结构化工作流，不追求完美辩论。

范围：

1. 三个角色 Agent 串行执行
2. 一个汇总 Agent 输出最终结论
3. 保存四份原始输出和结构化结果
4. 最终报告展示“三角色 + 汇总结论”
5. 继续生成 Markdown/HTML
6. 为每个阶段记录状态、耗时、解析结果和错误信息

不做：

1. 不做真正的角色互相反驳
2. 不做复杂权重系统
3. 不做多轮讨论
4. 不做复杂工作流引擎

### 8.2 产品化版本

目标：提升可解释性、可观测性和可维护性。

增强项：

1. 三 Agent 并发执行
2. 汇总 Agent 输出共识/分歧矩阵
3. 加入“证据强度”评分
4. 加入“仍待验证问题”
5. 前端显示多角色评分对比和分歧点
6. 支持角色结果查询和阶段重试
7. 区分粗粒度业务状态和细粒度进度阶段

### 8.3 高级版本

目标：形成真正的多智能体投研协作系统。

高级项：

1. 引入“反驳轮”
2. 汇总 Agent 二次追问子 Agent
3. 增加数据验证 Agent
4. 增加估值 Agent
5. 增加行业周期 Agent
6. 支持用户自定义 Agent 权重与风格
7. 支持更稳的任务调度与恢复机制

## 十、最终建议

### 建议结论

建议推进，并采用“结构化多 Agent + 汇总 Agent”的方案。

原因如下：

1. 与需求文档高度一致
2. 能显著提升报告的解释力和专业感
3. 对现有系统是增量改造，不是推倒重来
4. 后续扩展空间很大，能逐步演进为真正的投研协作系统

### 推荐路线

1. 先做 4 Agent MVP
2. 强制统一输出 Schema
3. 汇总 Agent 负责“共识、分歧、裁决”
4. 报告层再做 Markdown/HTML 编排
5. 真实数据链路优先补齐，否则多 Agent 价值会被削弱
6. 任务可靠性设计前置，不要等进入并发阶段再补

## 十一、补充说明（结合当前仓库现状）

结合当前仓库，需特别注意：

1. 当前 `analysis.py` 里仍存在 mock 数据和 mock ratios 逻辑
2. 这意味着现在更适合先完成多 Agent 骨架验证，而不是立刻追求结论准确率
3. 如果要正式上线，多 Agent 方案必须建立在更可靠的数据采集和数据验证基础上
4. 当前前端没有成熟测试链路，验证方式应以页面联调、`pnpm lint`、`pnpm build` 为主
5. 当前后台任务机制适合先验证流程，但不应默认视为最终生产方案

---

## 十二、上线前门槛建议

建议将以下项目作为正式上线前门槛：

1. 三个角色和汇总 Agent 的结构化输出解析成功率达到可接受水平
2. 关键财务字段来自真实数据，而非 mock 数据
3. 所有关键结论都能回溯到数据来源和日期
4. 单个 Agent 失败时可降级生成报告
5. 汇总阶段可单独重试
6. 前端可正确展示阶段进度、分角色结果和最终报告

---

## 十三、附录：一句话版本

### 产品话术版

将“三维合一分析”升级为“四 Agent 协作分析”：三位专业角色分别独立判断，再由汇总 Agent 统一裁决与编排，能显著提升报告的结构化、可解释性和专业可信度。

### 技术话术版

本方案本质是将单次 LLM 分析改造成“多角色结构化推理 + 汇总裁决”的工作流，工程上可基于现有 FastAPI 服务和报告生成链路做渐进式演进，但正式落地必须同步补齐任务可靠性、结构化解析约束和真实数据门槛。
