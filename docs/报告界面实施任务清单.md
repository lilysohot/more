# 报告界面前后端整合实施任务清单

来源文档：`docs/报告界排版.md`

本文档用于指导“智能投资分析报告界面”的落地开发，重点关注前端页面展示与后端业务数据的结构化整合。

## 1. 实施目标

- 将现有仅展示 `content_md/content_html` 的报告页，升级为结构化投研报告界面。
- 保留现有 Markdown/HTML 原文报告能力，避免破坏已有功能。
- 后端提供可直接消费的结构化报告数据，包括公司信息、财务指标、综合结论、Agent 分析、风险和证据来源。
- 前端按报告模板渲染投资结论、指标矩阵、三 Agent 分析、共识分歧、风险台账和证据架。
- 支持数据缺失、Agent 失败、报告未完成等降级状态。

## 2. 实施原则

- 先定义前后端数据契约，再开发页面组件。
- 新字段采用增量扩展方式，现有 `content_md/content_html` 保持不变。
- 前端优先做可降级展示：结构化数据存在时展示新模板，不存在时回退到原文报告。
- 每个阶段完成后都要保持应用可运行、可构建、可回退。

## 3. 阶段任务清单

### 阶段一：契约与数据来源梳理

| 状态 | 任务 | 主要产出 | 验收标准 |
|---|---|---|---|
| [√] | 定义结构化报告 API 契约 | 后端 Schema 草案、前端 TypeScript 类型草案 | 字段覆盖公司信息、财务指标、综合结论、Agent 结果、证据、原文报告 |
| [√] | 梳理后端现有数据来源 | 字段来源映射表 | 每个前端模块都能对应到后端字段或明确标注暂缺 |
| [√] | 设计接口兼容方案 | `GET /analyses/{id}/report` 扩展方案 | 原有 `content_md/content_html` 不变，新结构化字段可选 |

**重点关注**

- `Analysis`、`Report`、`AgentRun`、财务数据和综合 Agent 输出之间的关联。
- 现有报告生成流程是否已经持久化足够的结构化数据。
- 哪些字段只能从 Markdown/HTML 中看到，后端实际没有结构化保存。

### 阶段二：后端结构化报告能力

| 状态 | 任务 | 主要产出 | 验收标准 |
|---|---|---|---|
| [√] | 新增结构化响应 Schema | 更新 `backend/app/schemas/analysis.py` | FastAPI OpenAPI 能看到新增字段 |
| [√] | 实现报告结构化组装服务 | 新增或扩展 report assembler 逻辑 | 能组装 `company/financials/synthesis/agents/original` |
| [√] | 整合 AgentRun 数据 | `agents[]` 结构化输出 | 芒格、产业、审计 Agent 至少包含 `name/title/score/summary/risks/evidence` |
| [√] | 整合综合结论数据 | `synthesis` 结构化输出 | 包含 `final_score/investment_decision/core_reasons/major_risks/consensus/disagreements` |

- 第三方数据源字段可能缺失，后端应返回 `null` 或空数组，而不是伪造 `0`。
- 错误、降级、缺失数据需要显式表达，避免前端猜测。

**阶段验收**

## 7. 风险与应对

| 风险 | 影响 | 应对 |
|---|---|---|
| 后端没有持久化完整 Agent 结构化结果 | 前端无法展示 Agent Triad、证据架 | 先扩展保存逻辑或从 AgentRun 中解析结构化 JSON |
| 现有接口只返回 HTML/Markdown | 新模板无法按字段渲染 | 扩展报告响应，保留原字段兼容 |
| 第三方数据源字段缺失 | 指标矩阵出现大量空值 | 后端返回数据完整度，前端明确展示暂无数据 |
| Tushare 权限或频率限制导致财务接口不可用 | 真实报告可能出现财务字段 0/20 覆盖 | 后端按实际 `financials` 重新计算 `missing_financial_fields`，前端按覆盖率显示低置信度和“暂无数据” |
| HTML 注入存在安全边界 | 原文报告渲染风险 | 将 HTML 限制在附录区域，必要时净化或 iframe 沙箱 |
| 移动端 MainLayout 侧边栏限制 | 小屏阅读体验差 | 报告页内部使用响应式布局，必要时优化 MainLayout |
