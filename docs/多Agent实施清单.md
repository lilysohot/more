# 多 Agent 架构审定后的实施清单

**文档版本**：v1.0  
**编写日期**：2026-04-23

---

## 一、目标

本清单用于把以下两份文档收敛成可执行的落地方案：

1. `docs/多Agent方案报告.md`
2. `docs/多Agent技术设计文档.md`

补充说明：

1. `Phase 1` 的任务级拆分详见 `docs/多Agent Phase1开发任务清单.md`

本清单默认遵循以下原则：

1. 先验证多 Agent 工作流，再提升结论质量
2. 先保证结构化输出稳定，再增强辩论复杂度
3. 先保证任务可观测、可降级、可恢复，再做并发和高级能力

---

## 二、总体分期

### Phase 1：工作流 MVP

目标：

1. 跑通 3 个角色 Agent + 1 个汇总 Agent 的完整链路
2. 输出可稳定解析的结构化结果
3. 可生成分角色报告
4. 可记录每个 Agent 的运行信息

上线定位：

1. 内部验证
2. 工作流正确性验证
3. 不承诺高准确率投资结论

### Phase 2：产品化与可观测性

目标：

1. 完成前后端产品化闭环
2. 支持角色结果查询、共识分歧展示和阶段重试
3. 提升状态管理、解析稳定性和故障可观测性

上线定位：

1. 小范围可用
2. 可用于用户体验验证
3. 仍需持续观察数据质量和模型稳定性

### Phase 3：可靠性与质量提升

目标：

1. 提升真实数据质量
2. 升级任务执行可靠性
3. 支持并发执行和更强的多 Agent 协同能力

上线定位：

1. 可作为正式能力持续迭代
2. 进入质量优化和规模化阶段

---

## 三、Phase 1 实施清单

### 3.1 后端基础改造

1. 新增 `backend/app/services/agents/` 目录
2. 实现 `BaseAgent`
3. 实现 `munger_agent.py`
4. 实现 `industry_agent.py`
5. 实现 `audit_agent.py`
6. 实现 `synthesis_agent.py`
7. 实现 `orchestrator.py`
8. 在 `analysis.py` 中接入 `AgentOrchestrator`

### 3.2 结构化输出约束

1. 定义 `AgentContext`
2. 定义 `AgentResult`
3. 定义 `SynthesisResult`
4. 强制各 Agent 输出严格 JSON
5. 增加服务端 schema 校验
6. 增加一次解析失败后的修复重试
7. 保存原始输出和结构化结果

### 3.3 运行记录与持久化

1. 设计并落地 `agent_runs` 表或等价持久化结构
2. 记录每个 Agent 的状态、耗时、错误和原始输出
3. 为汇总 Agent 单独记录执行结果
4. 保持现有 `reports` 表继续承载最终 Markdown/HTML

### 3.4 报告生成

1. 将报告生成器从“单次长文本插入”升级为“基于结构化结果编排”
2. 报告包含三角色摘要
3. 报告包含汇总结论
4. 报告包含数据不足提示和失败视角提示

### 3.5 Phase 1 验收标准

1. 单次分析可生成 4 个 Agent 结果
2. 至少大部分角色输出能稳定结构化解析
3. 最终报告可展示三角色和汇总结论
4. 单个 Agent 失败时系统可降级输出报告
5. 每个 Agent 的原始输出和错误信息可追踪

### 3.6 Phase 1 明确不做

1. 不做多轮互相反驳
2. 不做用户自定义权重
3. 不做复杂工作流引擎
4. 不做完整生产级任务调度迁移

---

## 四、Phase 2 实施清单

### 4.1 接口与状态产品化

1. 将分析状态拆为 `status` 和 `progress_stage`
2. 扩展 `GET /api/v1/analyses/{analysis_id}/progress`
3. 新增 `GET /api/v1/analyses/{analysis_id}/agents`
4. 新增 `GET /api/v1/analyses/{analysis_id}/agents/{role}`
5. 新增汇总阶段重试接口

### 4.2 前端展示升级

1. 首页进度条支持细粒度阶段文案
2. 报告页展示三角色评分卡片
3. 报告页展示共识点和分歧点
4. 报告页展示数据质量提示
5. 报告页展示角色失败或缺失提示

### 4.3 解析与观测增强

1. 统计各 Agent 解析成功率
2. 统计各 Agent 耗时和失败率
3. 记录 `prompt_version`、`schema_version`、`model_name`
4. 区分调用失败、解析失败、业务失败

### 4.4 Phase 2 验收标准

1. 用户可查看分角色结果
2. 用户可看到清晰的阶段进度
3. 汇总阶段支持单独重试
4. 系统可追踪解析失败和调用失败的差异
5. 报告页具备基础可解释性展示能力

---

## 五、Phase 3 实施清单

### 5.1 真实数据质量提升

1. 替换或补强 mock 数据链路
2. 确保核心财务字段来自真实数据源
3. 为关键证据补齐来源、日期、类型
4. 建立数据不足和低置信度降级规则

### 5.2 任务执行可靠性升级

1. 评估并引入更稳的任务执行机制
2. 支持阶段级恢复和重试
3. 支持服务重启后的任务恢复或人工重驱动
4. 为超时、重试次数和失败原因建立统一规范

### 5.3 能力增强

1. 三角色并发执行
2. 引入反驳轮或冲突复核轮
3. 增加数据验证 Agent
4. 增加估值 Agent 或行业周期 Agent
5. 支持权重配置和策略调优

### 5.4 Phase 3 验收标准

1. 真实数据覆盖关键分析字段
2. 长链路任务可恢复、可重试、可观测
3. 并发执行后系统仍能稳定产出结构化结果
4. 汇总结果与证据链之间具备更强一致性

---

## 六、阶段门槛

### 从 Phase 1 进入 Phase 2 的门槛

1. 4 Agent 链路已跑通
2. 结构化解析成功率达到可接受水平
3. 报告生成已基于结构化结果
4. `agent_runs` 或等价记录已可用于排查问题

### 从 Phase 2 进入 Phase 3 的门槛

1. 前端已完成分角色展示和进度展示
2. 汇总重试和错误归因能力已具备
3. 数据来源可追溯能力已初步建立
4. 现有后台任务机制的限制已成为真实瓶颈，而不是理论担忧

---

## 七、优先级建议

建议按以下优先级推进：

1. P0：结构化 Schema、Agent 基类、编排器、运行记录
2. P0：报告生成器升级为消费结构化结果
3. P1：状态分层、分角色查询接口、前端分角色展示
4. P1：汇总重试、解析失败修复、观测指标
5. P2：真实数据增强、并发执行、任务执行机制升级
6. P3：反驳轮、更多 Agent、权重系统

---

## 八、当前建议的下一步动作

如果进入实施阶段，建议直接按以下顺序开工：

1. 先定义 `schemas.py` 中的三个核心结构
2. 再实现 `BaseAgent` 和 `AgentOrchestrator`
3. 再改造 `analysis.py` 接入多 Agent 流程
4. 再实现 `agent_runs` 持久化
5. 最后改造报告生成和前端展示
