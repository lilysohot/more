# 多 Agent 实施计划与排期

**文档版本**：v1.0  
**编写日期**：2026-04-23

---

## 一、目标

将当前单次 LLM 分析流程，升级为：

1. 芒格 Agent
2. 产业专家 Agent
3. 审计专家 Agent
4. 汇总 Agent

并最终生成结构化 Markdown / HTML 报告。

---

## 二、实施范围

本期建议范围：

1. 后端多 Agent 编排
2. 统一输出 Schema
3. 汇总 Agent 裁决与报告素材输出
4. Markdown / HTML 报告模板升级
5. 前端报告页结构升级

本期不建议范围：

1. 多轮 Agent 相互辩论
2. 用户自定义 Agent 权重
3. 完整工作流引擎
4. 复杂权限分层审阅流程

---

## 三、任务拆解

### 阶段 1：方案冻结

目标：确认输出口径与实现边界。

任务：

1. 明确 4 个 Agent 的职责边界
2. 确认统一 JSON Schema
3. 确认最终报告结构
4. 确认评分规则和分数区间定义

交付物：

1. 方案报告
2. 技术设计文档
3. Prompt 初稿

### 阶段 2：后端基础实现

目标：打通多 Agent 最小链路。

任务：

1. 新建 `agents/` 模块
2. 实现 Agent 基类
3. 实现 3 个角色 Agent
4. 实现汇总 Agent
5. 实现 `orchestrator.py`
6. 接入现有 `AnalysisService`

交付物：

1. 四个 Agent 可运行
2. 可输出结构化结果
3. 分析任务状态可追踪

### 阶段 3：报告生成升级

目标：将多 Agent 结果产出为统一报告。

任务：

1. 升级 Markdown 模板
2. 升级 HTML 模板
3. 增加共识与分歧展示
4. 增加角色评分区块

交付物：

1. 新版 Markdown 报告
2. 新版 HTML 报告

### 阶段 4：前端展示升级

目标：支持用户查看三角色与汇总结论。

任务：

1. 升级报告页布局
2. 增加角色评分展示
3. 增加共识/分歧模块
4. 增加风险提示模块

交付物：

1. 新版报告详情页
2. 更清晰的分析结果展示

### 阶段 5：质量与性能优化

目标：提升稳定性与成本控制。

任务：

1. 三角色并发执行
2. 增加超时与重试
3. 增加日志和失败降级
4. 优化 Prompt 长度与 Token 成本

交付物：

1. 性能优化版多 Agent 流程
2. 错误恢复能力

## 五、依赖关系

实施顺序建议：

1. 先定 Schema
2. 再定 Prompt
3. 再写 Agent 模块
4. 再接现有分析服务
5. 再升级报告生成
6. 最后升级前端展示

原因：

1. Schema 是后端、前端、报告模板的共同契约
2. Prompt 依赖角色定义和输出结构
3. 前端展示依赖后端最终输出格式稳定

---

## 六、风险与缓解

### 6.1 数据链路尚未真实完成

现状：当前仓库分析流程仍包含 mock 数据。

风险：多 Agent 结果会建立在不完整数据上。

缓解：

1. 先将多 Agent 作为编排验证版本
2. 同步推进真实数据采集
3. 报告中强制标注数据完整性

### 6.2 响应时间增长

风险：4 次模型调用可能导致用户等待时间明显上升。

缓解：

1. MVP 先接受较长耗时
2. 后续并发执行三角色
3. 减少角色返回长度

### 6.3 汇总结论不稳定

风险：汇总 Agent 可能受三份输入风格影响。

缓解：

1. 强制结构化输入
2. 限定汇总规则
3. 引入评分规则和权重

---

## 七、验收标准

### MVP 验收

1. 创建分析任务后，能成功执行四个 Agent
2. 每个 Agent 有明确状态与结果
3. 汇总 Agent 能生成最终结论
4. 报告可正常展示三角色与汇总内容
5. 报告可下载为 Markdown / HTML

### 增强版验收

1. 三角色可并发执行
2. 支持查看中间角色结果
3. 报告中包含共识与分歧模块
4. 失败时可降级生成报告
