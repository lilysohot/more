# 智能投资分析报告界面排版初版模板

## 1. 目标定位

本页面用于承载单次投资分析任务的最终报告，用户从首页提交公司名称或股票代码后，系统完成数据采集、财务比率计算、多 Agent 分析、综合汇总和报告生成，最终进入 `/report/:analysisId` 查看结果。

界面目标不是把 Markdown/HTML 直接贴出来，而是把“投资结论、证据、分歧、风险、数据来源”组织成一个可快速扫读、可深入展开、可导出的投研工作台。

## 2. 设计方向

### 2.1 视觉概念：暗色投研战情室 + 纸质研报层

整体采用“专业投研终端”的深色基底，叠加“象牙色研报纸张”的主体阅读层，形成数据终端和正式报告之间的对比。用户第一眼应记住的是：顶部像交易室大屏，正文像一份被精密标注的机构研报。

### 2.2 视觉关键词

- **基底**：深墨绿、近黑蓝、低饱和石墨色。
- **主体**：象牙纸白、淡米色卡片，避免普通纯白后台感。
- **强调**：铜金色用于结论与重点，风险使用克制的朱砂红，数据来源和置信度使用冷青色。
- **质感**：细网格、轻微噪点、证券终端式刻度线、报告纸张阴影。
- **排版**：大号结论、窄栏摘要、密集指标矩阵、可折叠证据层。

### 2.3 字体建议

- 中文标题：`Noto Serif SC` 或 `Source Han Serif SC`，强化正式研报感。
- 英文/数字：`IBM Plex Mono` 或 `JetBrains Mono`，用于指标、股票代码、评分和日期。
- 正文：`Noto Sans SC`，保证中文长段阅读稳定。
- 避免默认 `Arial`、纯系统字体和过度圆润的互联网产品感。

## 3. 当前业务流映射

### 3.1 用户路径

1. 用户在首页输入公司名称或股票代码。
2. 前端调用 `POST /api/v1/analyses` 创建分析任务。
3. 后端异步执行：标的识别、数据采集、比率计算、芒格 Agent、产业 Agent、审计 Agent、综合 Agent、报告生成。
4. 前端轮询 `/api/v1/analyses/{analysis_id}/progress`。
5. 完成后跳转 `/report/:analysisId`。
6. 报告页调用 `/api/v1/analyses/{analysis_id}/report` 获取 `content_md` 和 `content_html`。

### 3.2 当前报告页约束

- 现有前端只拿到 Markdown 和 HTML 字符串，无法直接按字段渲染精细模块。
- 后端实际拥有更丰富的数据：公司数据、财务比率、多 Agent 结果、综合结论、证据、风险、红旗、待确认问题。
- 初版界面可以先规划为 React 原生结构化报告模板，短期保留 HTML/Markdown 预览作为“原文报告”Tab。

## 5. 初版界面模板

### 5.1 Report Command Header

**目的**：让用户确认“这份报告分析的是谁、基于什么数据、何时生成”。

**字段来源**：
- `company_name`
- `stock_code` / `ts_code`
- `exchange`
- `industry`
- `data_source`
- `data_date`
- `report.created_at`
- `analysis.status`

**布局草案**：

```text
[智能投资分析师]

贵州茅台  600519.SH      食品饮料 / 白酒      数据源：东方财富 + Tushare
生成时间：2026-04-27 14:30  数据日期：2026-04-26  状态：分析完成

[返回] [下载 Markdown] [下载 HTML] [复制分享链接]
```

**视觉处理**：
- 深色横幅承载股票代码、交易所和状态。
- 股票代码使用等宽字体，像终端标签。
- 数据来源使用细小冷青色 Tag，避免喧宾夺主。

### 5.2 Hero Verdict：投资结论首屏

**目的**：用户进入页面后 5 秒内获得结论。

**字段来源**：
- `synthesis.final_score`
- `synthesis.investment_decision`
- `synthesis.core_reasons`
- `synthesis.major_risks`
- `synthesis.insufficient_data`

**布局草案**：

```text
┌───────────────────────────┬──────────────────────────────────┐
│ FINAL SCORE               │ 投资结论                           │
│ 7.6 / 10                  │ 谨慎增持 / 持续跟踪                 │
│ 铜金色环形评分             │ 数据充分性：中等                    │
├───────────────────────────┴──────────────────────────────────┤
│ 核心理由                                                      │
│ 1. 护城河仍具韧性，但增长斜率放缓。                            │
│ 2. 现金流质量高于利润表表现，安全边际取决于估值回落。            │
│ 3. 行业周期与库存压力仍需跟踪。                                │
├──────────────────────────────────────────────────────────────┤
│ 主要风险：估值压缩 / 渠道库存 / 需求恢复不及预期 / 数据缺口       │
└──────────────────────────────────────────────────────────────┘
```

**交互规则**：
- `final_score >= 8`：结论区偏绿色。
- `6 <= final_score < 8`：结论区偏铜金色。
- `final_score < 6`：结论区偏朱砂红。
- 数据不足时，在评分旁展示“低置信度”标识，避免制造确定性。

### 5.3 Company Profile：公司画像与业务本质

**目的**：在指标之前解释公司是什么、靠什么赚钱、处于什么产业位置。

**字段来源**：
- `synthesis.company_profile`
- `company_data.industry`
- `company_data.exchange`
- `company_data.market_cap`
- `company_data.revenue`
- `company_data.net_profit`

**模块内容**：
- 一句话业务本质。
- 所属行业、交易所、市值、收入、利润。
- 数据覆盖状态：行情、估值、财务、偿债、现金流。

**布局草案**：

```text
业务本质
这家公司通过高品牌势能和渠道控制获得超额毛利，核心变量是需求稳定性和价格纪律。

画像标签
[白酒] [高毛利] [强现金流] [估值敏感] [数据覆盖：财务/估值/现金流]
```

### 5.4 Metric Matrix：核心财务与估值矩阵

**目的**：把可量化指标变成“快速判断质量、估值、风险”的仪表盘。

**字段来源**：
- `revenue`
- `net_profit`
- `gross_margin`
- `net_margin`
- `roe`
- `roa`
- `market_cap`
- `pe_ratio`
- `pb_ratio`
- `ps_ratio`
- `asset_liability_ratio`
- `debt_to_equity`
- `current_ratio`
- `quick_ratio`
- `operating_cash_flow`
- `operating_cash_flow_to_net_profit`

**分组**：
- 规模：营业收入、净利润、市值。
- 盈利：毛利率、净利率、ROE、ROA。
- 估值：PE、PB、PS、收盘价。
- 偿债：资产负债率、流动比率、速动比率、产权比率。
- 质量：经营现金流、经营现金流/净利润。

**卡片模板**：

```text
┌──────────────┐
│ ROE          │
│ 18.4%        │
│ 盈利能力     │
│ 同组判断：强 │
└──────────────┘
```

**缺失数据处理**：
- 不显示 `0` 伪值；统一显示 `暂无数据`。
- 悬浮提示说明：`该指标未从当前数据源返回`。
- 模块顶部显示数据完整度：`已覆盖 16 / 22 项`。

### 5.5 Visual Insight：图表区

**目的**：让关键财务质量和估值风险一眼可比较。

**建议图表**：
- 盈利能力雷达图：毛利率、净利率、ROE、ROA、现金流质量。
- 估值条形图：PE、PB、PS。
- 偿债风险刻度：资产负债率、流动比率、速动比率。

**技术建议**：
- 前端依赖已包含 ECharts，React 原生模板建议使用 ECharts。
- 不建议继续在后端 HTML 中通过 CDN 引入 Chart.js。

### 5.6 Agent Triad：三维 Agent 分析

**目的**：展示系统核心差异化能力，即从芒格、产业、审计三个角度交叉验证投资判断。

**字段来源**：
- `agent_runs`
- `AgentResult.summary`
- `AgentResult.score`
- `AgentResult.thesis`
- `AgentResult.positives`
- `AgentResult.risks`
- `AgentResult.red_flags`
- `AgentResult.questions`
- `AgentResult.evidence`

**三卡片结构**：

```text
┌───────────────┬───────────────┬───────────────┐
│ 芒格视角       │ 产业视角       │ 审计视角       │
│ Score 8.1     │ Score 7.2     │ Score 6.8     │
│ 商业本质       │ 价值链位置     │ 利润质量       │
│ 护城河         │ 周期与供需     │ 现金流质量     │
│ 安全边际       │ 竞争格局       │ 会计红旗       │
│ [展开证据]     │ [展开证据]     │ [展开证据]     │
└───────────────┴───────────────┴───────────────┘
```

**每张卡片默认展示**：
- 评分。
- 一句话摘要。
- 3 条正向判断。
- 3 条主要风险。
- 是否存在红旗。

**展开后展示**：
- 完整 thesis。
- 证据列表。
- 待确认问题。
- 数据不足说明。

### 5.7 Consensus & Disagreement：共识、分歧与决策依据

**目的**：把多 Agent 输出从“多段文字”转成可决策的冲突矩阵。

**字段来源**：
- `synthesis.consensus`
- `synthesis.disagreements`
- `synthesis.core_reasons`
- `synthesis.major_risks`

**布局草案**：

```text
┌───────────────────────────┬──────────────────────────────┐
│ 共识                       │ 分歧                          │
│ 1. 品牌和渠道仍是核心资产   │ 1. 当前估值是否提供安全边际    │
│ 2. 现金流质量较强           │ 2. 行业需求恢复速度            │
│ 3. 监管和周期风险需跟踪     │ 3. 利润率能否维持              │
└───────────────────────────┴──────────────────────────────┘

决策依据
最终结论不是单一 Agent 投票结果，而是综合商业质量、产业周期、财务可靠性和估值安全边际后的判断。
```

### 5.8 Risk Ledger：风险台账

**目的**：让用户知道报告最担心什么，以及哪些问题需要后续尽调。

**字段来源**：
- `synthesis.major_risks`
- `AgentResult.risks`
- `AgentResult.red_flags`
- `AgentResult.questions`
- `AgentResult.insufficient_data`

**风险分级**：
- 一级风险：直接影响投资结论。
- 二级风险：影响估值或持仓节奏。
- 数据风险：当前数据源不足或证据置信度低。

**模板**：

```text
风险等级   风险事项             影响范围       后续跟踪
高         估值压缩             安全边际       等待估值回落或盈利上修
中         行业需求恢复不确定   增长假设       跟踪季度收入与渠道库存
中         现金流样本不足       财务质量       补充现金流量表历史数据
```

### 5.9 Evidence Shelf：证据与数据来源

**目的**：降低“AI 直接下结论”的黑箱感，让报告可追溯。

**字段来源**：
- `EvidenceItem.item`
- `EvidenceItem.source`
- `EvidenceItem.source_type`
- `EvidenceItem.source_date`
- `EvidenceItem.confidence`
- `EvidenceItem.excerpt`
- `company_data.data_source`
- `company_data.data_date`

**展示规则**：
- 默认折叠，按 Agent 或主题分组。
- 每条证据展示来源、日期、置信度、摘录。
- 明确标注“数据源事实”和“LLM 推理判断”的区别。

### 5.10 Original Report：原文报告

**目的**：兼容现有后端输出，保留完整 Markdown/HTML 报告查看与下载能力。

**当前可直接落地**：
- HTML 预览：`dangerouslySetInnerHTML` 渲染 `content_html`。
- Markdown 预览：`ReactMarkdown` 渲染 `content_md`。
- 下载 Markdown。
- 下载 HTML。
- 复制分享链接。

**建议位置**：
- 不再作为页面首屏。
- 放在结构化模块之后，作为“原文 / 附录”Tab。

## 6. 页面状态设计

### 6.1 Loading

- 顶部显示公司名和任务 ID。
- 主区域展示分析流程刻度：数据采集、比率计算、芒格、产业、审计、综合、报告生成。
- 使用深色骨架屏，不使用普通白底 Skeleton。

### 6.2 Empty

- 当报告不存在时显示：`报告尚未生成或已被删除`。
- 提供返回首页重新分析按钮。

### 6.3 Failed

- 显示失败阶段、错误信息、可重试入口。
- 如果已有部分 Agent 结果，允许展示“部分分析结果”。

### 6.4 Partial / Degraded

- Agent 失败或数据不足时，模块不留空。
- 使用 `数据不足`、`该视角未完成`、`证据缺失` 等明确提示。

## 10. 页面文案模板

### 顶部说明

```text
本报告由智能投资分析师基于公开数据、财务比率与多 Agent 推理生成。结论仅供研究参考，不构成投资建议。
```

### 数据不足提示

```text
部分关键数据未从当前数据源返回，本模块结论置信度较低。建议结合公司公告、年报和第三方行情终端继续核验。
```

### 风险免责声明

```text
投资有风险，入市需谨慎。模型输出可能受数据延迟、数据缺失、口径差异和推理偏差影响，请勿将本报告作为唯一投资依据。
```
