# 前端验证规范

## 适用范围

- `frontend/` 所有代码改动后的验证

## 本项目的固定事实

- 前端当前没有 `pnpm test` 脚本。
- `frontend/tests/` 目录当前为空，不能把它当成已建立的自动化测试体系。
- 前端可执行验证命令是：
  - `pnpm lint`
  - `pnpm build`
- `pnpm build` 已经包含 `tsc` 类型检查。

## 必做命令

1. 从 `frontend/` 目录运行 `pnpm lint`。
2. 再运行 `pnpm build`。
3. 不要在文档、说明或交付结论里写 `pnpm test`，因为本项目当前并不存在这个脚本。

## 手工验证清单

按改动范围至少覆盖对应场景：

1. 鉴权或布局改动：
   - 未登录访问是否回到 `/login`
   - 登录成功后是否进入首页
   - 退出登录后是否清空状态并跳回登录页
2. 分析首页改动：
   - 能否发起分析
   - 轮询进度时页面是否稳定
   - 完成后是否跳转报告页
3. API 配置页改动：
   - 列表加载
   - 新建/编辑/删除
   - 设为默认
   - 连通性测试反馈
4. 报告页改动：
   - 加载态
   - 分析未完成/失败时的失败态
   - 有结构化数据时的工作台展示
   - 无结构化数据时的原文降级展示
   - Markdown/HTML 下载和分享按钮
5. 个人中心改动：
   - 统计信息加载
   - 历史记录加载
   - 修改用户名
6. 样式改动：
   - 桌面端与移动端都能正常显示
   - 不破坏报告工作台现有视觉层次

## 与后端联动的额外要求

1. 如果前端改动依赖新的后端字段或新接口，不只做前端构建验证。
2. 需要在结论中说明依赖的后端前提，或补跑对应后端聚焦测试。

## 结果汇报规则

1. 明确写出实际运行过的命令。
2. 如果没做手工验证，要明确说明未验证的场景。
3. 如果因为环境或后端状态无法验证，要直接说明阻塞点。
